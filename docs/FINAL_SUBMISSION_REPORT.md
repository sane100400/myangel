# MyAngel 경진대회 최종 제출 보고서

작성일: 2026년 5월 14일  
서비스명: MyAngel  
라이브 서비스: https://ku-myangel.site  
주제: 한국어 사용자 중심의 AI 이미지 생성·부분 편집·저장·공유 웹 서비스

## 1. 프로젝트 개요

MyAngel은 사용자가 텍스트 프롬프트나 레퍼런스 이미지를 바탕으로 AI 이미지를 생성하고, 생성된 결과에서 원하는 부분만 다시 편집한 뒤, 완성본을 저장하고 공유할 수 있는 웹 기반 AI 이미지 스튜디오입니다.

기존 이미지 생성 서비스는 좋은 프롬프트를 작성하지 못하면 결과 품질이 크게 흔들리고, 부분 수정에서는 사용자가 바꾸고 싶은 대상이 모델에 정확히 전달되지 않는 문제가 있습니다. 예를 들어 "의자를 바꿔줘"라고 입력하면 모델이 어떤 의자를 의미하는지 추정해야 하므로, 원하지 않는 영역까지 변하거나 배경이 함께 달라질 수 있습니다.

MyAngel은 이 문제를 두 단계로 해결합니다.

1. 생성 단계에서는 한국어 프롬프트의 약한 표현을 감지하고 더 시각적인 문장으로 다듬습니다.
2. 편집 단계에서는 이미지 위에 추가·삭제·교체 마커를 직접 배치해, 모델이 수정해야 할 위치와 의도를 함께 받도록 했습니다.

결과적으로 MyAngel은 단순히 이미지 API를 호출하는 도구가 아니라, "어떤 이미지를 만들고 싶은지"와 "어느 부분을 고치고 싶은지"를 사용자가 더 정확하게 전달하도록 돕는 제품입니다.

## 2. 문제 정의

AI 이미지 생성 도구는 빠르게 발전했지만, 실제 사용자가 반복적으로 겪는 문제는 아직 남아 있습니다.

첫째, 프롬프트 작성 난이도입니다. 일반 사용자는 "예쁜", "감성적인", "고급스러운" 같은 추상적인 표현을 자주 사용합니다. 그러나 이미지 모델은 이런 표현을 매번 다르게 해석하므로 결과가 불안정해집니다.

둘째, 부분 편집의 위치 불확실성입니다. 자연어만으로 특정 영역을 수정하려고 하면 모델이 편집 대상을 잘못 이해하거나, 선택하지 않은 배경과 구도까지 바꾸는 경우가 생깁니다.

셋째, 생성 이후의 작업 흐름이 분리되어 있습니다. 이미지를 만들고, 다시 다운로드하고, 다른 편집 도구에 업로드하고, 결과를 따로 보관하는 과정은 사용자에게 번거롭습니다.

MyAngel은 이 세 가지 문제를 하나의 흐름으로 연결해 해결하는 것을 목표로 했습니다.

## 3. 핵심 해결 방식

### 3.1 한국어 프롬프트 강화

`/generate` 화면에서 사용자가 프롬프트를 입력하면, 약한 표현을 더 구체적인 시각 표현으로 바꿀 수 있습니다. 사용자는 추천 표현을 선택하고 문장을 다듬어, 이미지 모델에 더 명확한 입력을 보낼 수 있습니다.

예시:

```text
입력: 예쁜 카페의 감성적인 분위기
개선 방향: 노을빛이 들어오는 통창 카페, 따뜻한 우드 톤, 부드러운 그림자, 조용한 오후 분위기
```

이 기능은 프롬프트 작성 경험이 부족한 사용자도 더 안정적인 결과를 얻도록 돕습니다.

### 3.2 멀티모델 이미지 생성

MyAngel은 두 종류의 이미지 모델을 통합했습니다.

| 모델 | 적용 목적 | 지원 화질 |
|---|---|---|
| GPT Image 2 | 빠른 생성, 텍스트·로고·라벨 표현에 적합 | 1K, 2K |
| Nano Banana Pro | 스타일 표현과 고해상도 생성에 적합 | 1K, 2K, 4K |

사용자는 같은 프롬프트를 여러 모델에서 비교할 수 있습니다. 내부적으로는 모델별 지원 범위를 검증해, 지원하지 않는 조합이 API 요청으로 넘어가지 않도록 차단했습니다.

### 3.3 마커 기반 부분 편집

MyAngel의 가장 중요한 차별점은 마커 기반 편집입니다. 사용자는 `/edit` 화면에서 원형 마커를 배치하고 각 마커에 `추가`, `삭제`, `교체` 의도를 지정합니다.

지원하는 편집 방식은 다음과 같습니다.

| 편집 방식 | 설명 |
|---|---|
| 추가 | 선택한 위치에 새 요소를 넣음 |
| 삭제 | 선택한 위치의 요소를 제거하고 배경을 복원 |
| 교체 | 선택한 영역을 다른 객체나 스타일로 변경 |
| 전체 분위기·조명 조정 | 마커 없이 이미지 전체의 톤, 조명, 분위기를 조정 |

사용자는 단순히 원을 배치하지만, 내부에서는 이 마커 정보가 모델별 입력 형식으로 변환됩니다.

- Nano Banana Pro 경로: 원본 이미지, 마커 가이드, 그레이스케일 마스크, 레퍼런스 이미지를 함께 전달
- GPT Image 2 경로: 원본 이미지와 알파 마스크를 OpenAI 이미지 편집 API에 전달

이 방식은 자연어만 사용하는 편집보다 편집 대상의 위치를 명확히 전달할 수 있습니다.

### 3.4 저장·공유·크레딧 운영

생성 또는 편집 결과는 자동으로 MyPage에 저장됩니다. 사용자는 `/boards`에서 저장 이미지를 다시 열고, 편집기로 보내거나, Discover 갤러리에 공유할 수 있습니다.

또한 생성과 편집은 크레딧 기반으로 운영됩니다. 사용자는 예상 비용을 확인한 뒤 실행하며, 서버는 요청 전에 크레딧을 차감하고 전체 실패 시 환불합니다. 크레딧 차감은 Supabase RPC 단일 트랜잭션으로 처리해 동시 요청에서도 잔액이 음수로 내려가지 않도록 설계했습니다.

## 6. 개발 완성도

### 6.1 구현된 사용자 흐름

MyAngel은 대회 시연에서 다음 흐름을 처음부터 끝까지 보여줄 수 있습니다.

1. 사용자가 `/generate`에서 예시 프롬프트를 선택하거나 직접 프롬프트를 입력합니다.
2. 프롬프트 강화 기능으로 문장을 더 구체화합니다.
3. 모델, 화질, 이미지 비율, 생성 개수를 선택하고 이미지를 생성합니다.
4. 생성 결과는 자동으로 MyPage에 저장됩니다.
5. 결과 이미지를 편집기로 보내 원하는 영역에 마커를 배치합니다.
6. 마커별로 추가·삭제·교체 의도를 지정하고 필요하면 레퍼런스 이미지를 연결합니다.
7. 편집 결과를 다시 저장하고, Discover에 공유합니다.

이 흐름은 단일 기능 데모가 아니라 실제 사용자가 이미지를 만들고 고치고 보관하는 전체 사용 과정을 포함합니다.

### 6.3 시스템 아키텍처

MyAngel은 단일 화면형 데모가 아니라, 프론트엔드·API 서버·AI 모델·DB·스토리지가 연결된 제품형 구조로 구현했습니다.

```text
사용자 브라우저
  ├─ /generate: 프롬프트 강화, 레퍼런스 업로드, 모델/화질/비율 선택
  ├─ /edit: 베이스 이미지, 마커 좌표, 레퍼런스, 전역 분위기 조정 입력
  └─ /boards, /discover: 저장 이미지 관리와 공개 공유

Next.js API Layer
  ├─ Origin/Auth/Rate limit/이미지 검증
  ├─ 크레딧 비용 계산 및 deduct_credits RPC 호출
  ├─ OpenAI/Gemini 모델별 요청 어댑터
  ├─ SSE로 stage, image, image_failed, done 이벤트 전송
  └─ 실패 시 refund_credits RPC 호출

Supabase
  ├─ Auth: 이메일/Google 로그인
  ├─ Postgres: user_generations, discover_images, credit_ledger, pricing_config
  ├─ Storage: user-generations(private), shared-images(public)
  └─ RLS: 개인 저장 이미지는 본인만 접근, Discover는 공개 조회
```

프론트엔드는 이미지 결과를 기다리는 동안 화면이 멈추지 않도록 `fetch + ReadableStream` 기반 SSE 클라이언트를 사용합니다. 서버는 각 이미지 생성/편집이 끝나는 즉시 `image` 이벤트를 보내고, 실패한 개별 후보는 `image_failed`로 분리합니다. 따라서 4장을 요청했을 때 일부 결과가 먼저 도착하면 사용자는 바로 확인할 수 있고, 모든 요청이 실패한 경우에만 크레딧을 환불합니다.

### 6.4 주요 API 흐름

#### Generate 파이프라인

1. 클라이언트가 프롬프트, 모델, 화질, 비율, 개수, 레퍼런스 이미지를 `/api/generate`로 전송합니다.
2. 서버가 Origin, Supabase 세션, 모델·화질 지원 범위, 레퍼런스 이미지 개수와 매직 바이트를 검증합니다.
3. `pricing_config` 기준으로 비용을 계산하고 `deduct_credits` RPC를 idempotency key와 함께 호출합니다.
4. 모델에 따라 요청을 분기합니다.
   - GPT Image 2: `buildGptImage2GenerateRequest` 또는 `buildGptImage2EditRequest`로 허용 필드만 포함한 payload 생성
   - Nano Banana Pro: Google GenAI `generateContent`에 프롬프트와 레퍼런스 이미지 전달
5. 모델 응답 이미지를 `sharp`로 선택한 비율과 화질의 출력 크기에 맞게 정규화합니다.
6. 성공 이미지는 SSE `image` 이벤트로 즉시 전송하고, 전체 실패 시 `refund_credits`를 실행합니다.
7. 클라이언트는 도착한 결과를 `/api/saved-images`로 저장해 MyPage에서 다시 열 수 있게 합니다.

#### Edit 파이프라인

1. 클라이언트가 베이스 이미지, 마커 배열, 레퍼런스 이미지, 전체 분위기·조명 조정, 모델과 화질을 `/api/edit`로 전송합니다.
2. `validateEditRequest`가 base64 이미지, MIME, 매직 바이트, 마커 좌표, 마커 개수, 레퍼런스 개수, `refIndex`, 모델·화질 조합을 검증합니다.
3. 서버가 편집 비용을 차감하고 SSE 스트림을 엽니다.
4. `runMarkerEdit`가 원본 이미지 크기를 읽고, 모델별 입력 형식으로 변환합니다.
5. 편집 결과를 원본 이미지 비율에 맞게 후처리한 뒤 SSE로 후보 이미지를 순차 전송합니다.
6. 클라이언트는 성공 결과를 자동 저장하고, 사용자는 즉시 재편집·다운로드·Discover 공유를 할 수 있습니다.

| `src/lib/prompt-enhancer.ts` | 프롬프트 약한 표현 감지, 언어 판별, 결과 캐싱 |
| `src/lib/prompt-output-safety.ts` | 모델 응답에 시스템 지시문이 섞이는 경우 필터링 |
| `src/lib/editor-transfer.ts` | 생성/마이페이지 이미지의 편집기 이동 처리 |
| `src/lib/discover-store.ts` | Discover 이미지 DB 조회, Storage 공개 URL, 업로드/삭제 |
| `src/lib/sse.ts`, `src/lib/sse-client.ts` | 서버/클라이언트 SSE 스트리밍 처리 |

### 6.6 데이터 모델과 저장 전략

이미지는 크기가 크고 접근 권한이 서로 달라, DB와 Storage를 분리해 설계했습니다.

| 테이블/버킷 | 목적 | 접근 정책 |
|---|---|---|
| `user_generations` | 사용자의 생성·편집 결과 메타데이터 저장 | RLS로 본인만 조회·삭제 |
| `user-generations` Storage | 개인 저장 이미지 원본 보관 | private bucket, signed URL로 접근 |
| `discover_images` | 공개 공유 이미지의 제목, 프롬프트, 작성자 스냅샷, 생성 시각 저장 | 공개 조회, 작성자만 수정·삭제 |
| `shared-images` Storage | Discover 공개 이미지와 썸네일 보관 | public read, 작성자 경로에만 업로드/삭제 |
| `user_credits`, `credit_ledger` | 크레딧 잔액과 거래 원장 | 본인 조회, 쓰기는 service-role RPC |
| `pricing_config` | 생성·편집 단가와 가입 보너스 설정 | 공개 조회, 운영자가 DB에서 조정 |
| `error_logs` | 클라이언트·서버 오류 기록 | 본인 오류 조회 가능, 서버가 기록 |

공유 시에는 원본 이미지를 그대로 노출하지 않고 `sharp`로 WebP 변환과 썸네일 생성을 수행합니다. Discover 상세 페이지는 `created_at`, `width`, `height`를 함께 표시해 심사 중에도 생성 시각과 이미지 메타데이터를 확인할 수 있습니다.

## 7. 기술적 차별성

### 7.1 자연어 편집의 한계를 보완한 마커 프로토콜

이미지 부분 편집에서 핵심은 "무엇을 바꿀 것인가"뿐 아니라 "어디를 바꿀 것인가"입니다. MyAngel은 이 위치 정보를 원형 마커와 마스크로 명시합니다.

마커는 다음 구조로 서버에 전달됩니다.

```ts
type MarkerOp = "replace" | "add" | "remove";

interface MarkerSpec {
  id: string;
  op: MarkerOp;
  circle: {
    cx: number; // 0~1 정규화된 x 좌표
    cy: number; // 0~1 정규화된 y 좌표
    r: number;  // min(width, height) 기준 반지름
  };
  refIndex?: number;
  note?: string;
}
```

좌표를 픽셀 값이 아니라 0~1 정규화 값으로 저장하기 때문에, 화면에 표시된 이미지 크기와 실제 원본 해상도가 달라도 같은 위치를 안정적으로 복원할 수 있습니다. 서버에서는 `sharp`로 원본 이미지의 실제 `width`, `height`를 읽고, 정규화 좌표를 실제 픽셀 좌표로 변환한 뒤 마스크를 생성합니다.

Nano Banana Pro에는 그레이스케일 마스크를 전달합니다. 마스크 값은 보존 영역과 편집 의도를 구분합니다.

| 값 | 의미 |
|---:|---|
| 0 | 보존 |
| 64 | 삭제 |
| 128 | 추가 |
| 255 | 교체 |

GPT Image 2에는 알파 마스크를 전달해 투명 영역을 편집 대상으로 지정합니다. 모델별 입력 계약을 분리했기 때문에 사용자는 같은 UI를 쓰면서도 각 모델에 맞는 편집 입력을 사용할 수 있습니다.

마스크 가장자리에는 feather 처리를 적용했습니다. 원형 마커가 그대로 결과물에 드러나지 않도록 선택 영역보다 약간 넓은 soft influence area를 만들고, 실제 중심부는 명확하게 유지합니다. 또한 마커 중심은 이미지 모서리까지 이동할 수 있도록 클램핑 로직을 따로 두어, 모서리 영역을 편집하지 못하는 문제를 줄였습니다.

### 7.2 GPT Image 2 요청 계약 검증

GPT Image 2는 지원하는 화질·크기·요청 필드가 제한되어 있습니다. MyAngel은 지원하지 않는 조합을 UI와 API 양쪽에서 차단하고, GPT Image 2 요청에는 불필요한 `quality` 파라미터를 보내지 않도록 별도 builder와 테스트를 두었습니다.

이 구조는 단순한 API 호출보다 안정적입니다. 모델별 제약이 바뀌어도 요청 생성 함수를 중심으로 검증할 수 있습니다.

구체적으로 GPT Image 2 경로는 다음 제약을 강제합니다.

| 항목 | 구현 방식 |
|---|---|
| 지원 size 제한 | `1024x1024`, `1536x1024`, `1024x1536`만 허용 |
| 4K 차단 | 모델 지원표에서 GPT Image 2는 1K/2K만 허용 |
| 요청 필드 제한 | generate는 `model`, `prompt`, `n`, `size`만 허용 |
| edit 요청 필드 제한 | edit는 `model`, `image`, `mask`, `prompt`, `n`, `size`만 허용 |
| 출력 후처리 | OpenAI 응답을 앱의 1K/2K 출력 규격으로 `sharp` 리사이즈 |
| 마스크 크기 일치 | OpenAI 편집 입력의 base, reference, mask를 같은 PNG canvas로 정규화 |

특히 편집 API에서는 mask 이미지 크기가 base 이미지와 정확히 같아야 합니다. MyAngel은 업로드된 WebP/JPEG/PNG를 모두 PNG로 normalize하고, 레퍼런스 이미지도 같은 캔버스 크기에 맞춘 뒤 OpenAI에 전달합니다. 이 처리는 이미지 형식과 회전 메타데이터 차이 때문에 발생할 수 있는 `mask size does not match image size` 오류를 예방합니다.

### 7.3 프롬프트 강화와 모델 출력 안전장치

프롬프트 강화는 단순히 LLM에게 "잘 써줘"라고 요청하는 방식이 아니라, 약한 표현을 식별하고 교체 가능한 표현을 제안하는 구조입니다.

1. `detectPromptLanguage`가 한국어/영어 비율을 기준으로 입력 언어를 판단합니다.
2. Gemini가 원문에 실제로 존재하는 약한 표현과 대안을 JSON으로 반환합니다.
3. `scoreSpan`이 추상 형용사, 시각 어휘, 길이, 명사 구체성 등을 기준으로 후보 표현을 다시 점수화합니다.
4. 원문에서 위치를 찾을 수 없는 span은 제거합니다.
5. 같은 프롬프트 재분석은 LRU 캐시로 재사용합니다.

또한 LLM 응답에는 가끔 지시문, JSON 스키마, 시스템 메시지성 텍스트가 섞일 수 있습니다. 이를 막기 위해 `prompt-output-safety` 모듈에서 다음 처리를 수행합니다.

- 코드펜스나 JSON wrapper에서 실제 JSON만 추출
- `규칙:`, `JSON 형식:`, `system:`, `contents`, `thinkingConfig` 같은 지시문 패턴 필터링
- 추천 문구와 설명의 최대 길이 제한
- 문장 다듬기 결과가 지시문으로 판단되면 사용자의 수정 문장으로 fallback

이 방어는 프롬프트 강화 UI에 모델의 시스템 메시지가 노출되는 문제를 줄이고, 사용자 입력창에 지시문이 섞여 들어가는 것을 차단합니다.

### 7.4 운영 가능한 크레딧 시스템

크레딧은 단순히 화면에 숫자를 표시하는 수준이 아니라 서버 트랜잭션으로 처리됩니다.

- 요청 전 예상 비용 계산
- idempotency key 기반 중복 차감 방지
- `FOR UPDATE` lock 기반 동시 차감 직렬화
- 전체 실패 시 환불 ledger 생성
- 가격표를 `pricing_config`에서 관리

대회 이후 실제 서비스로 확장할 때도 과금과 비용 통제를 이어갈 수 있는 구조입니다.

### 7.5 보안 설계

AI 이미지 생성 서비스는 외부 API 비용이 발생하므로 인증과 남용 방지가 중요합니다. MyAngel은 다음 방어를 구현했습니다.

| 위협 | 대응 |
|---|---|
| 무인증 API 호출 | mutating endpoint에서 Supabase 사용자 확인 |
| CSRF | Origin 헤더 검증 |
| IDOR | Supabase RLS와 명시적 `user_id` 필터 |
| 이미지 위조 | base64 형식, MIME, 매직 바이트 검증 |
| SSRF | 허용 호스트 기반 URL 검증 |
| 과도한 요청 | 사용자/IP 기반 rate limit |
| 동시 차감 오류 | Postgres RPC 트랜잭션과 row lock |

이미지 입력은 단순 MIME 문자열만 믿지 않고 파일 시그니처를 확인합니다. `data:image/png`라고 주장해도 PNG 매직 바이트가 없으면 거부합니다. 외부 URL을 fetch해야 하는 경우에는 HTTPS와 허용 호스트 목록을 검사해 SSRF 위험을 줄였습니다.

### 7.6 실시간 처리와 장애 대응

이미지 모델 호출은 수십 초 이상 걸릴 수 있으므로, 일반 JSON 응답만 사용하면 사용자가 요청이 멈춘 것으로 느낄 수 있습니다. MyAngel은 생성과 편집 API에서 SSE를 사용합니다.

| 이벤트 | 의미 |
|---|---|
| `stage` | 시작, 모델 호출 중 등 현재 단계 |
| `heartbeat` | 장시간 요청 중 연결 유지 |
| `image` | 후보 이미지 1장 성공 |
| `image_failed` | 후보 이미지 1장 실패 |
| `done` | 요청 완료, 성공/실패 개수 전달 |
| `error` | 전체 실패 또는 서버 오류 |

이 구조 덕분에 이미지 4장을 요청했을 때 1장이 먼저 성공하면 즉시 화면에 표시할 수 있습니다. 모든 후보가 실패하면 크레딧을 환불하고, 일부만 실패한 경우에는 성공한 결과를 보여주면서 실패 정보를 함께 전달합니다.

## 8. 디자인 및 사용자 친화성

MyAngel의 UI는 사용자가 "생성 → 편집 → 저장/공유" 흐름을 빠르게 이해하도록 구성했습니다.

### 8.1 Generate 화면

Generate 화면은 프롬프트 입력을 중심에 두고, 예시 프롬프트, 레퍼런스 이미지, 모델 선택, 캔버스 비율, 화질, 생성 개수를 순서대로 배치했습니다. 예시 프롬프트는 짧은 칩으로 보이지만, 클릭하면 더 구체적인 2줄 프롬프트가 입력되어 초보자도 바로 결과를 만들 수 있습니다.

### 8.2 Edit 화면

Edit 화면은 이미지 위에 직접 마커를 놓는 방식입니다. 사용자는 추가·삭제·교체 버튼을 선택하고 원하는 위치를 지정합니다. 복잡한 모델 입력이나 마스크 파일을 직접 만들 필요가 없습니다.

### 8.3 MyPage와 Discover

MyPage는 개인 작업 보관함 역할을 합니다. 저장된 이미지를 다시 편집하거나 다운로드하고, 여러 이미지를 복수 선택해 삭제할 수 있습니다. Discover는 공유 결과를 모아 보여주는 공개 갤러리로, 대회 시연에서 결과물이 서비스 안에서 순환되는 모습을 보여줄 수 있습니다.

### 8.4 반응형 고려

주요 화면은 PC와 모바일에서 모두 사용할 수 있도록 반응형 레이아웃으로 구성했습니다. 편집 화면은 모바일에서 하단 고정 실행 버튼을 제공해 긴 화면에서도 핵심 액션에 접근할 수 있습니다.

## 9. 평가 기준별 정리

### 9.1 기획: 문제 해결 목표와 적용 가능성

MyAngel의 목표는 AI 이미지 생성 과정에서 발생하는 두 가지 핵심 불확실성, 즉 프롬프트의 애매함과 부분 편집 위치의 모호함을 줄이는 것입니다. 이 문제는 디자이너뿐 아니라 발표 자료, 썸네일, 쇼핑몰 상품 이미지, 동아리 홍보물, SNS 콘텐츠를 만드는 일반 사용자에게도 자주 발생합니다.

서비스 확장성도 있습니다. 현재는 개인 사용자 중심의 크레딧 모델이지만, 이후에는 팀 단위 이미지 워크스페이스, 브랜드 템플릿, 편집 히스토리, 상업용 이미지 검수 기능으로 확장할 수 있습니다.

### 9.2 창의성 및 독창성

단순 이미지 생성 서비스와 달리 MyAngel은 마커 기반 부분 편집을 핵심 경험으로 제시합니다. 사용자가 원을 찍는 간단한 UI 뒤에서 모델별 마스크 입력을 생성하고, 추가·삭제·교체 의도를 함께 전달하는 구조는 프로젝트의 독창적인 지점입니다.

또한 한국어 프롬프트 강화, 멀티모델 선택, 크레딧 운영, 저장·공유 흐름을 하나의 제품으로 연결한 점이 차별점입니다.

### 9.3 개발 완성도

기획한 핵심 기능은 대부분 실제 서비스에서 동작합니다. 생성, 편집, 저장, 공유, 인증, 크레딧, 보안 검증, 배포까지 구현되어 있으며 라이브 URL에서 접근 가능합니다.

검증 결과:

- `npm test`: 15개 테스트 파일, 92개 테스트 통과
- 주요 경로 라이브 확인: `/`, `/generate`, `/edit`, `/boards`, `/discover` 모두 HTTP 200
- 서버 배포: GCP VM에서 systemd 서비스 `active`

### 9.4 기술적 우수성

프로젝트는 단순 CRUD 웹 앱을 넘어 외부 AI 모델, 이미지 마스크 처리, SSE 스트리밍, Supabase RLS, Storage, 크레딧 트랜잭션, 보안 검증을 함께 다룹니다.

특히 다음 구현은 기술적 난이도가 높습니다.

- 모델별 이미지 편집 입력 계약 분리
- 마커 좌표를 정규화하고 실제 이미지 크기에 맞게 마스크 렌더링

- 크레딧 차감의 동시성 제어
- 대용량 이미지 저장을 위한 Supabase Storage 연동
- 생성/편집 진행 상황을 SSE로 표시

|---|---|---|
| 모델 통합 | OpenAI GPT Image 2와 Google Nano Banana Pro를 같은 UI에서 선택 | 서로 다른 API 계약을 하나의 제품 경험으로 추상화 |
| 요청 안정성 | 모델별 지원 화질, size, payload key를 코드와 테스트로 제한 | 잘못된 옵션이 외부 API 비용으로 이어지는 문제 방지 |
| 부분 편집 | 원형 마커를 정규화 좌표로 저장하고 실제 이미지 크기에 맞춰 마스크 렌더링 | UI 좌표와 모델 입력 간 불일치 해결 |
| 이미지 처리 | `sharp`로 EXIF 회전, PNG 변환, WebP 변환, 썸네일 생성, 출력 크기 정규화 | 이미지 형식·비율·해상도 차이를 서버에서 일관 처리 |
| 실시간 UX | SSE로 후보 이미지별 완료 이벤트 전송 | 긴 AI 작업에서도 진행 상황을 즉시 피드백 |
| 결제성 로직 | Postgres RPC, row lock, idempotency key, ledger 기반 환불 | 동시 요청·재시도·실패 상황에서도 크레딧 정합성 유지 |
| 보안 | Origin 검증, RLS, 매직 바이트 검증, SSRF 호스트 제한, rate limit | 외부 API 비용과 개인 이미지 데이터를 보호 |
| 운영성 | systemd 서비스, nginx reverse proxy, Supabase Storage, 오류 로그 | 로컬 데모가 아닌 라이브 운영 가능한 구조 |

또한 구현 과정에서 실제 오류를 해결하며 안정성을 높였습니다. 예를 들어 GPT Image 2 편집에서 mask와 image 크기가 다르면 API가 400 오류를 반환하기 때문에, 베이스 이미지와 레퍼런스 이미지를 같은 PNG canvas로 정규화하는 처리를 추가했습니다. 프롬프트 강화에서는 모델 출력에 시스템 지시문이 섞이는 문제를 서버에서 필터링하도록 보완했습니다. 이런 개선은 단순 기능 구현을 넘어 실제 사용자 오류와 모델 제약을 반영한 엔지니어링입니다.

### 9.5 디자인 및 UX/UI

화면은 기능별로 분리되어 있지만 사용 흐름은 끊기지 않도록 설계했습니다. Generate에서 만든 이미지를 바로 Edit로 보내고, 결과는 MyPage에 자동 저장되며, 필요한 경우 Discover로 공유됩니다.

또한 모델과 화질 옵션처럼 실수하기 쉬운 부분에는 UI 안내와 비활성 상태를 적용했습니다. 예를 들어 GPT Image 2에서 지원하지 않는 4K 조합은 선택할 수 없도록 처리했습니다.

## 12. 한계와 개선 계획

현재 버전은 대회 제출과 시연이 가능한 수준으로 구현되어 있지만, 서비스 확장을 위해 개선할 부분도 명확합니다.

| 항목 | 현재 상태 | 개선 계획 |
|---|---|---|
| 부분 환불 | 전체 실패 시 환불, 일부 성공/실패는 전액 환불하지 않음 | 성공 장수 기준 부분 환불 정책 추가 |

| 편집 히스토리 | 결과 저장 중심 | 버전 비교와 되돌리기 기능 추가 |

| 결과 품질 관리 | 후보 이미지 선택 중심 | 자동 품질 점수, 실패 원인 분석 추가 |

## 13. 최종 요약

MyAngel은 한국어 사용자에게 맞춘 AI 이미지 생성·편집 스튜디오입니다. 프로젝트의 핵심 가치는 사용자가 원하는 이미지를 더 적은 시행착오로 만들게 하는 것입니다.

이 목표를 위해 프롬프트 강화, 멀티모델 생성, 마커 기반 부분 편집, 저장·공유, 크레딧 운영을 하나의 제품 흐름으로 구현했습니다. 특히 마커 기반 편집은 자연어만으로는 불명확했던 편집 위치를 이미지 위에서 직접 지정하게 해, 실제 사용성과 기술적 차별성을 동시에 보여주는 기능입니다.

현재 서비스는 배포되어 있으며, 생성·편집·저장·공유의 핵심 흐름을 실제로 시연할 수 있습니다. 따라서 MyAngel은 아이디어 단계에 머무르지 않고, 대회 평가 기준인 기획, 독창성, 개발 완성도, 기술적 우수성, 디자인 및 UX를 모두 보여줄 수 있는 완성형 제출물입니다.
